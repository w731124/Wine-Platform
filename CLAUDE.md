# 葡萄酒學習平台｜專案核心精省憲法

本憲法為新對話開始時唯一必須遵守的最高框架，取代任何說明書文件。憲法本身不含、也不應含任何現況資訊（檔案清單、資料筆數、功能完成度）——這類資訊一律以當次對話中使用者提供或已上傳的實際程式碼為準，不得憑空假設。

## 一、鎖定慣例（政策性規定，非程式碼可自證，需憲法明文保留）
- 視覺：背景 #FAF8F5、酒紅 #5C061C、金色 #C5A880、標題字型 Cinzel serif，淺色基調，不可更動。
- 命名：所有地名與分類／標籤欄位一律採「English(中文)」並列格式。
- 回應語言：一律使用繁體中文。

## 二、架構鐵律
- 模組化為不可逆決策，嚴禁以任何理由將程式碼重組回單一檔案。
- 分層傾向：新增邏輯時盡量將資料處理與 DOM 渲染區隔，既有混合處不強行拆分重構。
- 非必要不做全檔重構，以達成需求的最小範圍為原則。

## 三、技術死線
- 動態圖表更新須確認具備銷毀舊實例邏輯（如 Chart.js 的 destroy），防止記憶體洩漏。
- 新增樣式優先使用 Tailwind CSS 語意化 class；既有行內樣式處維持現狀，不主動新增新的行內樣式。

## 四、對話與輸出精省條款
- 非經明確要求，不主動讀取或掃描存放大宗靜態資料的區塊（不論副檔名）。
- 一律輸出精準的程式碼片段並標註檔案路徑與插入位置，不輸出完整檔案。
- **預設不擴大稽核範圍**：任務過程中若意外發現與本次任務無關的其他問題，僅於報告後附註一行提及，不展開處理、不中斷或擴大原任務範圍，除非被明確要求。

## 五、執行前哨機制
收到修改指令後禁止直接輸出程式碼，須先以不超過兩句話回報：
1. 打算修改哪一個檔案的哪一段邏輯
2. 預計改動範圍與受影響的模組

## 六、工作流程規則
完成一個階段性功能後：
1. 將本次改動的關鍵決策與原因，追加寫入 DECISIONS.md
2. 執行 git commit，訊息需具體描述改了什麼、為什麼

**若對目標檔案的現行內容不確定，兩句話報告中必須明講這是假設，並要求使用者提供該段現行程式碼，不得逕行假設現況動工。**

回報後立即停止，等待明確回覆（如「確認」或「Go」）才可動工。
